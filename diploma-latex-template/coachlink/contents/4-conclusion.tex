\conclusion

В результате выполнения выпускной квалификационной работы была спроектирована и реализована backend-часть открытой микросервисной платформы CoachLink, предназначенной для автоматизации взаимодействия в системе «тренер-спортсмен». Разработанная система решает практическую проблему разрозненного обмена тренировочными планами и отчетами через мессенджеры, электронные таблицы и произвольные файлы. Вместо набора не связанных между собой каналов коммуникации платформа предоставляет единый backend-контур, в котором тренировочные задания, отчеты, уведомления и аналитические данные хранятся и обрабатываются в структурированном виде.

Основной результат работы состоит в создании серверной основы платформы. Мобильное приложение в рамках данной работы рассматривается только как внешний клиент REST API и не является результатом разработки автора. Такой фокус позволяет рассматривать систему именно с позиции backend-проектирования: разделения ответственности между сервисами, организации API, хранения данных, событийного взаимодействия, авторизации, тестирования и возможности дальнейшего развития.

\subsection*{Достижение цели работы}

Целью работы была разработка открытой микросервисной backend-платформы для стандартизации и автоматизации взаимодействия между тренерами и спортсменами. Данная цель была достигнута за счет проектирования и реализации набора независимых сервисов, каждый из которых отвечает за отдельную область предметной логики.

В платформе реализованы следующие ключевые возможности:

\begin{itemize}
\tightlist
\item
  регистрация и аутентификация пользователей с использованием JWT;
\item
  разделение пользователей по ролям тренера и спортсмена;
\item
  хранение пользовательских профилей;
\item
  оформление и обработка заявок на установление связи между тренером и спортсменом;
\item
  создание тренировочных групп;
\item
  создание тренировочных планов и шаблонов;
\item
  назначение тренировок отдельным спортсменам и группам;
\item
  отправка спортсменами стандартизированных отчетов о выполнении тренировок;
\item
  хранение статусов назначений и отчетов;
\item
  формирование уведомлений на основе событий системы;
\item
  агрегация тренировочных данных для аналитических представлений;
\item
  получение AI-рекомендаций по последним отчетам спортсмена.
\end{itemize}

Таким образом, backend-платформа охватывает полный цикл базового взаимодействия: от регистрации пользователя и установления связи между тренером и спортсменом до назначения тренировки, отправки отчета, уведомления участников и последующего анализа данных.

\subsection*{Решение поставленных задач}

В ходе выполнения работы были решены задачи, сформулированные во введении.

Первая задача заключалась в анализе предметной области и существующих способов организации тренировочного процесса. Было показано, что мессенджеры, таблицы, спортивные трекеры и закрытые платформы решают отдельные части проблемы, но не предоставляют одновременно единый формат отчетности, поддержку ролей тренера и спортсмена, групповые назначения, открытую архитектуру, API и возможность самостоятельного развертывания. На основе этого анализа были сформулированы требования к backend-платформе.

Вторая задача заключалась в проектировании архитектуры системы. Для решения этой задачи была выбрана микросервисная архитектура, в которой функциональность разделена между API Gateway, Auth Service, User Service, Training Service, Notification Service, Analytics Service и AI Service. Такое разделение позволяет развивать отдельные части системы независимо и снижает связанность между модулями. Для межсервисного взаимодействия используются REST API и событийная коммуникация через NATS JetStream.

Третья задача состояла в реализации сервисов платформы. В рамках backend-части были реализованы сервисы аутентификации, пользователей, тренировок, уведомлений, аналитики, AI-рекомендаций и API Gateway. Каждый сервис имеет собственную область ответственности и собственную схему данных либо, в случае stateless-сервисов, работает как вычислительный модуль поверх данных других сервисов.

Четвертая задача была связана с описанием публичного API и межсервисных контрактов. Платформа предоставляет REST API, сгруппированный по функциональным областям: аутентификация, пользователи, тренировки, уведомления, аналитика и AI-рекомендации. API Gateway выступает единой точкой входа и отвечает за проверку JWT, маршрутизацию запросов и передачу пользовательского контекста во внутренние сервисы.

Пятая задача заключалась в проверке работоспособности backend-части. Для проекта используются unit-тесты сервисного слоя, E2E smoke-сценарий и интеграционные тесты через API Gateway. Подтверждены успешные запуски unit-тестов сервисов, компиляции интеграционного тестового пакета, а также E2E и полного интеграционного прогона при поднятой платформе.

\subsection*{Практическая значимость}

Практическая значимость работы заключается в том, что разработанный backend может использоваться как основа для реального сервиса взаимодействия тренера и спортсмена. Платформа устраняет несколько типичных проблем, возникающих в тренировочном процессе.

Во-первых, тренер получает единый механизм назначения тренировок. Вместо отправки сообщений каждому спортсмену отдельно он может создавать планы, назначать их конкретным спортсменам или группам и отслеживать статусы выполнения.

Во-вторых, спортсмен получает единый формат отчета. Это особенно важно для последующего анализа, поскольку структурированные данные можно агрегировать, сравнивать и использовать для построения статистики. В мессенджерах данные обычно существуют в виде произвольного текста, фотографий или файлов, что затрудняет автоматическую обработку.

В-третьих, платформа формирует централизованную историю тренировочного процесса. Накопленные данные позволяют анализировать динамику нагрузки, регулярность выполнения тренировок и изменения показателей спортсмена.

В-четвертых, система уведомлений снижает вероятность пропуска важных событий. Пользователь может получать уведомления о новых заявках, назначениях, отчетах и других действиях, связанных с тренировочным процессом.

В-пятых, AI Service добавляет вспомогательный аналитический слой. Он не заменяет тренера и не принимает решений вместо него, но помогает сформировать рекомендации на основе последних отчетов спортсмена. Такой подход делает AI-модуль безопасным с точки зрения роли в системе: итоговое решение остается за специалистом.

\subsection*{Архитектурные результаты}

Одним из важных результатов работы является то, что платформа построена как набор независимых сервисов, а не как монолитное приложение. Это решение оправдано предметной областью, поскольку разные части системы имеют разные причины для изменения.

Auth Service отвечает за регистрацию, вход, обновление токенов и публикацию события \passthrough{\lstinline!coachlink.user.registered!}. Он изолирует логику безопасности и не смешивает ее с предметной логикой тренировок.

User Service отвечает за профили, заявки, связи между тренерами и спортсменами, группы и внутренние проверки доступа. Он является источником информации о социальных связях внутри платформы.

Training Service отвечает за тренировочные планы, шаблоны, назначения и отчеты. Это центральный предметный сервис, в котором сосредоточена основная логика тренировочного процесса.

Notification Service подписывается на события из NATS JetStream и создает уведомления для пользователей. Благодаря этому сервисы, в которых возникают события, не обязаны напрямую знать о способе доставки уведомлений.

Analytics Service выполняет stateless-агрегацию тренировочных данных. Он получает данные из Training Service и формирует summary, progress и coach overview без хранения собственной предметной базы.

AI Service предоставляет одну реализованную ручку рекомендаций по последним отчетам спортсмена. Он использует данные аналитики и генерацию через Ollama, оставаясь вспомогательным модулем.

API Gateway объединяет сервисы за единым публичным API, проверяет JWT, проксирует запросы и передает во внутренние сервисы пользовательский контекст.

Такое разделение повышает сопровождаемость системы: изменение механизма уведомлений не требует переписывания Training Service, изменение аналитики не влияет на регистрацию пользователей, а развитие AI-модуля может происходить отдельно от базового цикла назначения и отчетности.

\subsection*{Ограничения текущей версии}

Текущая версия платформы имеет осознанные ограничения, связанные с рамками дипломной работы и реалистичным объемом реализации.

Предметная реализация ориентирована на легкую атлетику. Это сделано потому, что автор занимается этим видом спорта и лучше понимает структуру тренировочных данных: дистанции, объем, интенсивность, самочувствие, комментарии по выполнению. При этом архитектура не ограничивает платформу только легкой атлетикой. Поддержка других видов спорта может быть реализована через развитие структуры тренировок, отчетов, шаблонов и sport-specific метрик.

Мобильное приложение не входит в зону ответственности данной работы. Backend предоставляет REST API, к которому могут подключаться разные клиенты: мобильное приложение, web-интерфейс, административные панели или внешние интеграции.

AI Service в текущей версии не является универсальной системой анализа тренировочного процесса. Он реализует один сценарий: формирование рекомендаций по последним отчетам конкретного спортсмена. Расширенные AI-сценарии, такие как комплексный анализ периода или сводка по группе для тренера, относятся к направлениям дальнейшего развития.

В текущем backend не реализованы production-механизмы уровня промышленной эксплуатации: rate limiting, Redis-кеширование, расширенная наблюдаемость, автоматическое масштабирование, полноценная отказоустойчивость и интеграции со спортивными устройствами. Эти направления относятся к дальнейшему развитию.

\subsection*{Направления дальнейшего развития}

Дальнейшее развитие платформы может идти по нескольким направлениям.

Первое направление - поддержка других видов спорта. Для этого необходимо расширить модель тренировочного плана и отчета так, чтобы она позволяла описывать не только беговые и легкоатлетические тренировки, но и силовые упражнения, игровые тренировки, плавание, велоспорт и другие виды активности. Важно сохранить общую часть модели: назначение, отчет, статус, комментарий, связь тренера и спортсмена, но добавить специализированные поля для отдельных спортивных дисциплин.

Второе направление - более детальная структура тренировок. В текущей версии тренировка может описываться как план и отчет, но в будущем можно выделить отдельные упражнения, интервалы, серии, повторения, зоны интенсивности, контрольные показатели и целевые метрики. Это позволит точнее сравнивать план и факт выполнения.

Третье направление - интеграции со спортивными устройствами и внешними сервисами. Платформа может получать данные из часов, пульсометров, GPS-трекеров и популярных спортивных экосистем. Это позволит автоматически заполнять часть отчета и снижать нагрузку на спортсмена.

Четвертое направление - развитие аналитики. Можно добавить анализ нагрузки по периодам, выявление пропусков, сравнение планового и фактического объема, оценку восстановления, предупреждения о резком росте нагрузки и индивидуальные графики прогресса.

Пятое направление - расширение AI-сценариев. После накопления большего объема данных можно реализовать дополнительные функции: сводки по группе, анализ динамики спортсмена за период, выявление потенциальных рисков, подготовку черновика комментария тренера. При этом AI должен оставаться вспомогательным инструментом, а не заменой профессионального решения тренера.

Шестое направление - подготовка backend к production-эксплуатации. Сюда относятся централизованное логирование, трассировка запросов, метрики, health checks, rate limiting, управление секретами, резервное копирование баз данных, мониторинг очередей и настройка CI/CD.

\subsection*{Итоговый вывод}

Разработанная backend-платформа CoachLink демонстрирует возможность построения открытой микросервисной системы для автоматизации тренировочного процесса. Система объединяет роли тренера и спортсмена, стандартизирует обмен тренировочными данными, поддерживает групповые сценарии, формирует уведомления и создает основу для аналитики.

Главный результат работы заключается не только в реализации отдельных API-методов, но и в создании расширяемой серверной архитектуры. Платформа может развиваться как в сторону более глубокого описания тренировок в легкой атлетике, так и в сторону поддержки других видов спорта. Благодаря открытой архитектуре и разделению сервисов проект может быть дополнен новыми клиентскими приложениями, интеграциями, аналитическими модулями и AI-сценариями без полного перепроектирования backend-части.

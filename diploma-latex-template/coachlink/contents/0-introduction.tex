\introduction

Современный тренировочный процесс всё чаще опирается не только на очное взаимодействие тренера и спортсмена, но и на постоянный обмен цифровыми данными. Тренер должен заранее формировать тренировочные задания, распределять их между спортсменами, учитывать индивидуальные особенности подготовки, получать обратную связь после выполнения тренировки и принимать решения о дальнейшем изменении нагрузки. При работе с одним спортсменом такие действия могут выполняться вручную и не требуют сложной информационной системы. Однако при увеличении количества спортсменов и появлении тренировочных групп ручной обмен сообщениями быстро становится неудобным и ненадёжным.

На практике взаимодействие тренера со спортсменами часто организуется с помощью мессенджеров, электронных таблиц и произвольных файлов. Тренер может отправлять задания в личные сообщения, групповые чаты или таблицы, а спортсмены присылают отчёты в удобном для них виде. Такой подход не требует внедрения специализированного программного обеспечения, но создаёт несколько существенных проблем.

Во-первых, отсутствует единый формат данных. Отчёты спортсменов отличаются по структуре, полноте и способу представления. Из-за этого тренеру приходится вручную приводить информацию к сопоставимому виду. Во-вторых, затрудняется поиск и анализ истории тренировок. Данные распределены по разным чатам, таблицам и файлам, поэтому их сложно агрегировать и использовать для оценки динамики подготовки. В-третьих, сообщения о тренировках смешиваются с личной и организационной перепиской. Тренер может пропустить важный отчёт или несвоевременно отреагировать на жалобу спортсмена. В-четвёртых, при работе с группами становится сложно контролировать, кому было назначено задание, кто его выполнил, кто отправил отчёт, а кто ещё нет.

Эти проблемы особенно заметны в циклических видах спорта, в которых важно не только получить отдельный отчёт, но и накопить историю структурированных данных. На основе такой истории можно рассчитывать сводные показатели, отслеживать прогресс, замечать признаки чрезмерной нагрузки и принимать более обоснованные решения при планировании следующих тренировок.

В связи с этим актуальной является задача разработки единой backend-платформы, которая автоматизирует взаимодействие тренера и спортсмена, стандартизирует формат тренировочных отчётов, обеспечивает хранение истории тренировок и предоставляет программные интерфейсы для внешних клиентских приложений. Такая платформа должна решать не только задачу обмена сообщениями, но и задачу организации предметной логики тренировочного процесса: ролей пользователей, связей между тренером и спортсменами, групп, планов, назначений, отчётов, уведомлений и аналитики.

В данной работе рассматривается backend-часть платформы CoachLink. Backend реализует серверную логику, хранение данных, публичные REST API, межсервисное взаимодействие и интеграцию с инфраструктурными компонентами. Мобильное приложение рассматривается только как внешний клиент, который обращается к backend через API Gateway. Разработка мобильного приложения не входит в область ответственности данной выпускной квалификационной работы.

Такое разграничение важно, поскольку основная инженерная задача данной работы заключается не в проектировании пользовательского интерфейса, а в создании устойчивой серверной основы для платформы. Именно backend определяет модель данных, правила доступа, жизненный цикл тренировочных заданий, структуру отчётов, механизм уведомлений и возможность дальнейшей аналитики.

Платформа CoachLink проектируется как расширяемое средство для автоматизации тренировочного процесса в разных видах спорта. Первая реализация ориентирована только на лёгкую атлетику — это сделано для упрощения разработки первого прототипа. При этом архитектура платформы остаётся универсальной: базовые сущности (пользователь, тренер, спортсмен, группа, план, назначение, отчёт, уведомление) не привязаны к конкретному виду спорта и в будущем могут быть дополнены специфическими метриками и шаблонами для других дисциплин.

Объектом исследования является процесс информационного взаимодействия между тренером и спортсменом при планировании, выполнении и анализе тренировок.

Предметом исследования является backend-платформа для автоматизации назначения тренировочных заданий, сбора стандартизированных отчётов, отправки уведомлений и анализа данных тренировочного процесса.

Целью выпускной квалификационной работы является разработка открытой микросервисной backend-платформы CoachLink для стандартизации и автоматизации взаимодействия в системе «тренер-спортсмен».

Для достижения поставленной цели необходимо решить следующие задачи:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  провести анализ существующих способов организации взаимодействия тренера и спортсмена;
\item
  выявить ограничения мессенджеров, электронных таблиц, спортивных трекеров и закрытых платформ при решении задачи автоматизации тренировочного процесса;
\item
  сформулировать функциональные и нефункциональные требования к backend-платформе;
\item
  спроектировать микросервисную архитектуру системы и определить ответственность каждого сервиса;
\item
  спроектировать модель данных для пользователей, связей, групп, тренировочных планов, назначений, отчётов и уведомлений;
\item
  реализовать сервис авторизации с поддержкой JWT access token и refresh token;
\item
  реализовать сервис пользователей, отвечающий за профили, связи тренер-спортсмен и тренировочные группы;
\item
  реализовать сервис тренировок, отвечающий за планы, назначения, шаблоны и отчёты;
\item
  реализовать сервис уведомлений, создающий уведомления на основе событий системы;
\item
  реализовать сервис аналитики, агрегирующий данные тренировочного процесса;
\item
  реализовать AI-сервис рекомендаций по последним отчётам спортсмена;
\item
  описать публичные API с помощью OpenAPI;
\item
  провести тестирование backend-компонентов и зафиксировать подтверждённые результаты.
\end{enumerate}

При выполнении работы используются методы системного анализа, объектно-ориентированного и компонентного проектирования, проектирования REST API, проектирования реляционных баз данных и тестирования программного обеспечения. Выбор микросервисной архитектуры соотносится с подходами к построению распределённых систем, описанными в работах по шаблонам микросервисов, проектированию независимых сервисов и переходу к cloud-native архитектурам~\cite{richardson-microservices,newman-microservices-2021,dragoni-microservices,balalaie-microservices-devops}.

В качестве основного языка реализации backend выбран Go~\cite{go-docs}. Для разработки HTTP API используется Echo~\cite{echo-docs}. Для хранения данных применяется PostgreSQL~\cite{postgres-docs}. Для работы с SQL-запросами используется \passthrough{\lstinline!sqlx!}, для миграций --- \passthrough{\lstinline!goose!}. Для аутентификации применяется JWT~\cite{jwt-rfc}, для хеширования паролей --- bcrypt. Асинхронное межсервисное взаимодействие реализовано через NATS JetStream~\cite{nats-docs}. Для локального развёртывания используется Docker Compose~\cite{docker-docs}. API документируется в формате OpenAPI~\cite{openapi-spec}. AI-рекомендации формируются через локальный LLM-сервер Ollama~\cite{ollama-docs}.

Теоретическая значимость работы заключается в проектировании модели backend-платформы, объединяющей микросервисный подход, событийное взаимодействие, стандартизированный сбор тренировочных данных и аналитическую обработку этих данных. Работа демонстрирует, как предметная область тренировочного процесса может быть представлена через набор независимых сервисов с чёткими зонами ответственности.

Практическая значимость заключается в создании открытой серверной основы для платформы CoachLink. Реализованный backend может использоваться как база для дальнейшего развития системы: подключения мобильного приложения, добавления новых видов спорта, расширения аналитики, интеграции со спортивными устройствами и развития AI-сценариев. Благодаря открытому API систему можно адаптировать под разные клиентские приложения и внешние интеграции.

Выпускная квалификационная работа состоит из введения, трёх глав, заключения и приложений.

В первой главе проводится аналитический обзор существующих подходов к организации взаимодействия тренера и спортсмена. Рассматриваются мессенджеры, электронные таблицы, спортивные трекеры и закрытые платформы. На основе сравнения формулируются требования к backend-платформе.

Во второй главе описывается проектирование CoachLink. Рассматриваются архитектура микросервисов, модель данных, межсервисное взаимодействие, ключевые пользовательские сценарии и выбор технологического стека.

В третьей главе описывается практическая реализация backend-сервисов и результаты тестирования. Отдельно рассматриваются API Gateway, Auth Service, User Service, Training Service, Notification Service, Analytics Service и AI Service.

В заключении подводятся итоги работы, оценивается достижение цели и формулируются направления дальнейшего развития платформы.

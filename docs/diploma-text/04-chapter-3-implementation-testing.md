# Глава 3. Реализация и тестирование

## 3.1 Общая структура реализации

Backend-платформа CoachLink реализована как набор Go-сервисов в монорепозитории. Каждый сервис имеет собственную структуру проекта, включающую точку входа, конфигурацию, HTTP-обработчики, бизнес-логику, модели данных и слой доступа к данным или внешним клиентам.

Основные директории backend:

| Директория | Назначение |
|---|---|
| `services/api-gateway` | Единая точка входа и reverse proxy |
| `services/auth-service` | Авторизация и управление токенами |
| `services/user-service` | Профили, связи и группы |
| `services/training-service` | Планы, назначения, отчёты, шаблоны |
| `services/notification-service` | Уведомления и FCM-токены |
| `services/analytics-service` | Аналитика тренировочных данных |
| `services/ai-service` | AI-рекомендации |
| `pkg/events` | Общие типы событий NATS |
| `deployments` | Docker Compose и инициализация БД |
| `tests/integration` | Интеграционные тесты через API Gateway |

Сервисы запускаются в Docker Compose вместе с PostgreSQL, NATS JetStream, Ollama и вспомогательными контейнерами. Для каждого сервиса предусмотрен Dockerfile. Stateful-сервисы применяют SQL-миграции при запуске.

## 3.2 API Gateway

API Gateway реализован как отдельный Go-сервис. Его основная задача — принять внешний HTTP-запрос и перенаправить его в нужный backend-сервис. Для этого используются префиксы маршрутов:

| Префикс | Целевой сервис |
|---|---|
| `/api/v1/auth/*` | Auth Service |
| `/api/v1/users/*` | User Service |
| `/api/v1/connections/*` | User Service |
| `/api/v1/groups/*` | User Service |
| `/api/v1/training/*` | Training Service |
| `/api/v1/notifications/*` | Notification Service |
| `/api/v1/analytics/*` | Analytics Service |
| `/api/v1/ai/*` | AI Service |

Gateway выполняет JWT-проверку для всех защищённых маршрутов. Исключениями являются регистрация, вход, обновление токена и health-check. После успешной проверки токена Gateway извлекает из claims идентификатор пользователя, роль и логин. Эти значения передаются дальше через заголовки:

```text
X-User-ID: <user_id>
X-User-Role: <coach|athlete>
X-User-Login: <login>
```

Такой механизм позволяет внутренним сервисам принимать решения о доступе без повторной проверки JWT. Например, Training Service проверяет, может ли текущий пользователь создать план, отправить отчёт или получить назначение.

Gateway также отвечает за CORS и health-check:

```text
GET /health
```

Важно отметить, что rate limiting в текущей реализации не рассматривается как реализованная функциональность. Его можно добавить в дальнейшем как middleware уровня Gateway.

## 3.3 Auth Service

Auth Service отвечает за регистрацию, вход и управление токенами. Сервис использует PostgreSQL-базу `auth_db` и таблицы `users` и `refresh_tokens`.

### 3.3.1 Регистрация

При регистрации пользователь передаёт логин, email, пароль, ФИО и роль. DTO запроса содержит ограничения:

- логин обязателен, длина от 3 до 50 символов;
- email должен иметь корректный формат;
- пароль обязателен, длина от 8 до 128 символов;
- ФИО обязательно;
- роль должна быть `coach` или `athlete`.

Дополнительно сервис проверяет формат логина регулярным выражением:

```text
^[a-zA-Z0-9-]+$
```

Это ограничение исключает пробелы, кириллицу и специальные символы в логине, что упрощает поиск и использование логина в API.

Пароль хешируется через bcrypt. В базе хранится только `password_hash`, исходный пароль не сохраняется. После создания пользователя сервис публикует событие `coachlink.user.registered`, содержащее идентификатор пользователя, логин, ФИО, email и роль.

### 3.3.2 Вход и токены

При входе сервис ищет пользователя по логину и сравнивает пароль с bcrypt-хешем. При успешной проверке создаётся пара токенов:

- access token — JWT с коротким сроком жизни;
- refresh token — opaque token с более длинным сроком жизни.

Access token содержит claims:

```json
{
  "sub": "uuid",
  "login": "athlete-login",
  "role": "athlete",
  "exp": 1234567890,
  "iat": 1234567890
}
```

Refresh token не хранится в открытом виде. Сервис вычисляет SHA-256 хеш токена и сохраняет только хеш. При refresh старый токен удаляется, затем создаётся новая пара access/refresh. При logout refresh token удаляется из базы.

### 3.3.3 Основные API Auth Service

| Метод | Endpoint | Назначение |
|---|---|---|
| `POST` | `/api/v1/auth/register` | Регистрация |
| `POST` | `/api/v1/auth/login` | Вход |
| `POST` | `/api/v1/auth/refresh` | Обновление access token |
| `POST` | `/api/v1/auth/logout` | Отзыв refresh token |

## 3.4 User Service

User Service хранит пользовательские профили, связи тренер-спортсмен и тренировочные группы. Он использует базу `user_db`.

### 3.4.1 Синхронизация профилей

Профиль пользователя создаётся не напрямую при регистрации, а через событие `coachlink.user.registered`. Auth Service публикует событие после успешной регистрации, а User Service подписан на него через NATS JetStream. При получении события User Service создаёт запись в таблице `user_profiles`.

Такое разделение позволяет Auth Service отвечать только за учётные данные и токены, а User Service — за доменную информацию, используемую в тренировочном процессе.

### 3.4.2 Поиск пользователей

Сервис предоставляет поиск по ФИО и логину. Поиск используется, например, когда спортсмен ищет тренера для отправки заявки или тренер ищет спортсмена. Минимальная длина поискового запроса — два символа. Это ограничение уменьшает количество слишком широких запросов.

Для ускорения поиска в базе включено расширение `pg_trgm`, а для полей имени и логина создаются trigram-индексы. Это подходит для поиска подстрок и автодополнения.

### 3.4.3 Заявки и связи

Связь тренера и спортсмена создаётся через заявку. Основной сценарий:

1. Спортсмен находит тренера.
2. Спортсмен отправляет заявку.
3. Тренер получает уведомление.
4. Тренер принимает или отклоняет заявку.
5. При принятии создаётся запись в `coach_athlete_relations`.

В первой версии у спортсмена может быть один тренер. Это ограничение задаётся уникальностью `athlete_id` в таблице связей. Такое решение упрощает логику доступа: Training Service и Analytics Service могут однозначно определять, какой тренер отвечает за спортсмена.

### 3.4.4 Тренировочные группы

Тренер может создавать тренировочные группы и добавлять туда своих спортсменов. Группа используется для массового назначения тренировочного плана. Перед добавлением спортсмена в группу сервис проверяет, есть ли связь между этим спортсменом и тренером.

User Service предоставляет следующие операции:

- создание группы;
- получение списка групп;
- получение деталей группы с участниками;
- обновление названия группы;
- удаление группы;
- добавление спортсмена;
- удаление спортсмена.

### 3.4.5 Внутренние API

User Service предоставляет внутренние API для других сервисов:

| Endpoint | Использование |
|---|---|
| `GET /internal/users/{userId}` | Получение профиля пользователя |
| `GET /internal/groups/{groupId}/members` | Получение состава группы |
| `GET /internal/coach/{coachId}/has-athlete/{athleteId}` | Проверка связи тренера и спортсмена |

Эти API не предназначены для внешних клиентов. Они позволяют Training Service, Analytics Service и AI Service выполнять предметные проверки и получать необходимые данные.

## 3.5 Training Service

Training Service является центральным сервисом тренировочного процесса. Он использует базу `training_db`.

### 3.5.1 Тренировочные планы

Тренировочный план содержит название, описание, дату и идентификатор тренера. План может быть назначен:

- одному или нескольким спортсменам по списку `athlete_ids`;
- всем спортсменам группы через `group_id`;
- комбинации группы и отдельных спортсменов.

Если указан `group_id`, Training Service обращается к User Service и получает участников группы. Затем создаётся один тренировочный план и набор назначений для каждого спортсмена.

### 3.5.2 Назначения

Назначение представляет связь между тренировочным планом и конкретным спортсменом. В назначении хранятся:

- `plan_id`;
- `athlete_id`;
- `coach_id`;
- имя и логин спортсмена;
- имя и логин тренера;
- статус;
- время назначения;
- время выполнения;
- время архивации.

Денормализация имени и логина спортсмена используется для удобной фильтрации списков назначений без дополнительного запроса в User Service.

Статусы назначений:

| Статус | Значение |
|---|---|
| `assigned` | Задание назначено и ожидает отчёта |
| `completed` | Спортсмен отправил отчёт |
| `archived` | Тренер перенёс выполненное назначение в архив |

### 3.5.3 Просроченность задания

Просроченность вычисляется функцией `ComputeIsOverdue`. Задание считается просроченным, если:

- статус равен `assigned`;
- дата тренировки уже прошла.

Просроченность не хранится в базе данных как отдельное поле. Это уменьшает количество фоновых задач и исключает ситуацию, когда сохранённое значение устаревает.

### 3.5.4 Отчёты

Спортсмен может отправить отчёт только по своему назначению и только если назначение находится в статусе `assigned`. После создания отчёта статус назначения меняется на `completed`.

Формат отчёта:

```json
{
  "content": "Тренировка прошла хорошо, последние интервалы дались тяжело",
  "duration_minutes": 75,
  "perceived_effort": 7,
  "max_heart_rate": 188,
  "avg_heart_rate": 162,
  "distance_km": 12.4
}
```

Поля отчёта выбраны с учётом первой реализации для лёгкой атлетики. Они позволяют фиксировать как объективные показатели, так и субъективную оценку нагрузки.

### 3.5.5 Шаблоны тренировок

Тренер может создавать шаблоны тренировок. Шаблон содержит название и описание. Он используется для повторяющихся тренировок, чтобы тренеру не приходилось каждый раз вручную вводить одно и то же задание. При создании плана можно сохранить его как шаблон или использовать существующий шаблон как основу.

### 3.5.6 События Training Service

Training Service публикует события:

| Событие | Когда публикуется |
|---|---|
| `coachlink.training.assigned` | Создано назначение |
| `coachlink.training.deleted` | Назначение удалено |
| `coachlink.report.submitted` | Спортсмен отправил отчёт |

Эти события используются Notification Service для создания уведомлений.

## 3.6 Notification Service

Notification Service подписан на события NATS JetStream и создаёт уведомления в базе `notification_db`.

### 3.6.1 Обработка событий

Для каждого типа события создаётся отдельный durable consumer. Это позволяет сервису продолжить обработку после перезапуска и не потерять события.

Примеры соответствия событий и уведомлений:

| Событие | Получатель | Тип уведомления |
|---|---|---|
| `connection.requested` | Тренер | Новая заявка от спортсмена |
| `connection.accepted` | Спортсмен | Заявка принята |
| `training.assigned` | Спортсмен | Новое тренировочное задание |
| `report.submitted` | Тренер | Новый отчёт |
| `group.athlete_added` | Спортсмен | Добавление в группу |

Если уведомление успешно создано, сообщение подтверждается через `Ack`. Если произошла ошибка, сообщение отклоняется через `Nak`, чтобы оно могло быть обработано повторно.

### 3.6.2 API уведомлений

Notification Service предоставляет:

- получение списка уведомлений с пагинацией;
- фильтрацию по признаку прочтения;
- получение количества непрочитанных уведомлений;
- пометку одного уведомления прочитанным;
- пометку всех уведомлений прочитанными;
- регистрацию FCM-токена устройства.

FCM-интеграция реализована как опциональная. Если credentials не настроены, push-доставка отключается, но серверные уведомления продолжают работать. Это важно для локальной разработки и тестирования без внешних Firebase-настроек.

## 3.7 Analytics Service

Analytics Service не хранит собственные данные. Он получает отчёты и агрегаты из Training Service через внутренние HTTP API.

### 3.7.1 Сводка спортсмена

Сводка включает:

- количество отчётов;
- суммарную длительность;
- среднюю длительность;
- суммарную дистанцию;
- среднюю воспринимаемую нагрузку;
- средний пульс;
- максимальный пульс;
- количество назначений;
- процент выполнения.

Эти показатели позволяют тренеру оценить объём и регулярность тренировок спортсмена.

### 3.7.2 Прогресс по периодам

Analytics Service группирует отчёты по неделям или месяцам. Для каждого периода рассчитываются:

- количество отчётов;
- суммарная длительность;
- средняя нагрузка;
- средний пульс;
- суммарная дистанция.

Группировка по периодам позволяет видеть динамику тренировочного процесса, а не только итоговые значения за всё время.

### 3.7.3 Обзор тренера

Для тренера сервис формирует обзор по его спортсменам: количество спортсменов, назначений, отчётов и процент выполнения. Такой endpoint нужен для быстрого понимания общей картины по группе.

## 3.8 AI Service

AI Service предоставляет одну публичную ручку:

```text
POST /api/v1/ai/athletes/{id}/recommendations
```

### 3.8.1 Проверка доступа

Сервис доступен только пользователю с ролью `coach`. После получения запроса AI Service обращается к User Service и проверяет, связан ли указанный спортсмен с текущим тренером. Если связи нет, возвращается ошибка доступа.

### 3.8.2 Получение данных

После проверки доступа сервис получает из Analytics Service:

- сводную статистику спортсмена;
- последние отчёты спортсмена.

Если отчётов нет, сервис возвращает ошибку недостатка данных. Это необходимо, потому что AI-рекомендация должна опираться на фактическую тренировочную историю, а не на пустой контекст.

### 3.8.3 Формирование prompt

Prompt формируется на русском языке. Он содержит:

- общую статистику спортсмена;
- последние тренировки;
- длительность;
- дистанцию;
- субъективную нагрузку;
- показатели пульса;
- комментарии спортсмена;
- дополнительный контекст от тренера.

Количество отчётов ограничивается пятью. Типовой сценарий — рекомендации по последним 3-5 отчётам. Это снижает размер prompt и ускоряет генерацию.

### 3.8.4 Генерация через Ollama

AI Service обращается к Ollama API:

```text
POST /api/chat
```

Используется модель `gemma3:4b`. Ответ возвращается тренеру в виде структуры:

```json
{
  "athlete_id": "uuid",
  "type": "recommendations",
  "content": "Текст рекомендаций от LLM...",
  "generated_at": "2026-04-13T12:00:00Z",
  "model": "gemma3:4b"
}
```

AI-модуль не является заменой тренеру. Он предоставляет вспомогательный текст, который тренер может использовать при принятии решения о корректировке нагрузки.

## 3.10 Тестирование

### 3.10.1 Уровни тестирования

Для проверки backend используются несколько уровней тестирования.

Unit-тесты проверяют бизнес-логику отдельных сервисов без запуска всей инфраструктуры. Такой подход позволяет быстро проверять правила доступа, обработку ошибок и вычисления.

E2E smoke-тест проверяет основной пользовательский сценарий через HTTP API. Он нужен для быстрой проверки того, что основные сервисы работают совместно.

Интеграционные тесты проверяют endpoint-группы через API Gateway. Они ближе всего к поведению реального внешнего клиента, потому что запросы проходят через тот же входной слой, что и мобильное приложение.

### 3.10.2 Unit-тесты

Unit-тесты покрывают:

| Сервис | Что проверяется |
|---|---|
| `training-service` | Просроченность, доступ к заданиям, архивация, отправка отчётов, шаблоны |
| `notification-service` | Регистрация FCM-токена, пагинация, отметка прочтения, создание уведомления |
| `analytics-service` | Сводка, группировка по неделям/месяцам, сортировка, ошибки клиента |
| `ai-service` | Успешная генерация, недостаток данных, ошибки Ollama, лимит отчётов, prompt |

Команда запуска:

```bash
make test-unit
```

Фактически подтверждённый результат: команда `make test-unit` выполнена успешно. Прошли unit-тесты `ai-service`, `analytics-service`, `training-service` и `notification-service`.

### 3.10.3 Компиляция интеграционных тестов

Для проверки актуальности интеграционного тестового пакета используется компиляция тестового бинарника:

```bash
cd tests/integration
go test -c
```

Фактически подтверждённый результат: интеграционный пакет компилируется. Это подтверждает, что тестовые функции согласованы с текущим набором endpoint-ов, в том числе с единственной реализованной AI-ручкой рекомендаций.

### 3.10.4 E2E и интеграционные тесты

Для полного запуска E2E и интеграционных тестов необходимо поднять платформу:

```bash
make up
make test-e2e
make test-integration
```

Если эти команды не были выполнены в момент подготовки финального текста, в дипломе нельзя утверждать, что соответствующие тесты успешно пройдены. Допустимо указать, что тесты предусмотрены и описать их назначение, но результаты должны фиксироваться только после фактического запуска.

### 3.10.5 Таблица результатов

| Проверка | Статус | Комментарий |
|---|---|---|
| `make test-unit` | Успешно | Проверена бизнес-логика четырёх сервисов |
| `go test -c` в `tests/integration` | Успешно | Интеграционный пакет компилируется |
| `make test-e2e` | Не зафиксировано | Требует поднятой платформы |
| `make test-integration` | Не зафиксировано | Требует поднятой платформы |

## 3.11 Выводы по главе 3

В третьей главе была рассмотрена реализация backend-платформы CoachLink. Реализованы сервисы, обеспечивающие регистрацию и авторизацию пользователей, управление профилями и связями, тренировочные группы, планы, назначения, отчёты, уведомления, аналитику и AI-рекомендации.

Система построена вокруг единого API Gateway, который проверяет JWT и маршрутизирует запросы. Межсервисное взаимодействие реализовано через HTTP и NATS JetStream. Данные разделены по сервисам, что соответствует выбранной микросервисной архитектуре.

Проведённая проверка показывает, что ключевая бизнес-логика покрыта unit-тестами, а интеграционный тестовый пакет согласован с текущим API. Для финальной версии работы необходимо дополнительно выполнить E2E и полный интеграционный прогон при поднятой платформе и зафиксировать результаты.
